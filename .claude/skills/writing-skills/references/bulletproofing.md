## When This Applies

Skills that enforce a rule the agent will be tempted to skip: TDD, verification-before-completion, designing-before-coding, refusing-shortcuts, "do X before Y" gates. These skills carry compliance costs (time, effort, sunk-work) and will be rationalized away under pressure unless the rule is bulletproofed.

Skip these techniques for technique skills, pattern skills, or pure reference skills — they have no rule to violate.

## The Iron Law

```
NO SKILL WITHOUT A FAILING TEST FIRST
```

Applies to **new skills AND edits to existing skills**. Write skill before testing? Delete it. Edit skill without testing? Same violation.

**No exceptions:**
- Not for "simple additions"
- Not for "just adding a section"
- Not for "documentation updates"
- Don't keep untested changes as "reference"
- Don't "adapt" while running tests

The TDD discipline applies to documentation the same way it applies to code. See [pressure-testing.md](pressure-testing.md) for the RED-GREEN-REFACTOR cycle adapted to skill testing.

## Close Every Loophole Explicitly

Stating the rule isn't enough. Forbid specific workarounds the agent will reach for.

❌ Weak:
```markdown
Write code before test? Delete it.
```

✅ Bulletproof:
```markdown
Write code before test? Delete it. Start over.

**No exceptions:**
- Don't keep it as "reference"
- Don't "adapt" it while writing tests
- Don't look at it
- Delete means delete
```

The list of forbidden workarounds is generated by [pressure-testing.md](pressure-testing.md) — every rationalization the agent invents under pressure becomes a new loophole to close.

## Address Spirit-vs-Letter Arguments

Add a foundational principle near the top:

```markdown
**Violating the letter of the rules is violating the spirit of the rules.**
```

This single line cuts off an entire class of "I'm following the spirit, the letter doesn't matter" rationalizations. Without it, agents will route around any specific rule by claiming alignment with its purpose.

## Build a Rationalization Table

Capture every excuse from baseline testing. Each rationalization gets a row with the reality-check counter:

```markdown
| Excuse | Reality |
|--------|---------|
| "Too simple to test" | Simple code breaks. Test takes 30 seconds. |
| "I'll test after" | Tests passing immediately prove nothing. |
| "Tests after achieve same goals" | Tests-after = "what does this do?" Tests-first = "what should this do?" |
| "Keep as reference, write tests first" | You'll adapt it. That's testing-after. Delete means delete. |
| "Already manually tested it" | Manual ≠ automated. Manual ≠ regression-proof. |
```

This table is generated by RED-phase testing — see [pressure-testing.md](pressure-testing.md). Don't pre-populate it from imagination; populate it from observed failures.

## Create a Red Flags List

Make self-checking trivial. Agents about to rationalize should hit a STOP marker:

```markdown
## Red Flags — STOP and Start Over

- Code before test
- "I already manually tested it"
- "Tests after achieve the same purpose"
- "It's about spirit not ritual"
- "This is different because..."
- "Just this once"
- "Being pragmatic vs dogmatic"

**All of these mean: Delete code. Start over with TDD.**
```

The phrases are exact agent vocabulary captured during pressure testing. Generic warnings ("don't cheat") don't work; specific phrases do.

## Encode Violation Symptoms in the Description

The description triggers the skill. Add symptoms of *being about to violate the rule* so the skill loads at the right moment:

```yaml
description: Use when implementing any feature or bugfix, before writing implementation code. Use when tempted to test after, when manual testing seems faster, or when 'just this once' rationalizations appear.
```

Triggering on violation-precursors makes the skill arrive before the agent commits to the wrong path.

## Persuasion Principles

Bulletproofing leans on specific persuasion techniques. See [persuasion-principles.md](persuasion-principles.md) for the research foundation (Cialdini 2021, Meincke et al. 2025) and which principles fit which skill type. For discipline skills: authority + commitment + social proof. Avoid liking and reciprocity (they create sycophancy and false debt).

## Common Rationalizations for Skipping Testing

Frequently observed when agents are asked to skip testing the skill itself:

| Excuse | Reality |
|--------|---------|
| "Skill is obviously clear" | Clear to you ≠ clear to other agents. Test it. |
| "It's just a reference" | References can have gaps, unclear sections. Test retrieval. |
| "Testing is overkill" | Untested skills have issues. Always. 15 min testing saves hours. |
| "I'll test if problems emerge" | Problems = agents can't use skill. Test BEFORE deploying. |
| "Too tedious to test" | Testing is less tedious than debugging a bad skill in production. |
| "I'm confident it's good" | Overconfidence guarantees issues. Test anyway. |
| "Academic review is enough" | Reading ≠ using. Test application scenarios. |
| "No time to test" | Deploying untested skill wastes more time fixing it later. |

All resolve to: test before deploying.

## Real-World Impact

From applying TDD to the TDD-skill itself (one documented case):
- 6 RED-GREEN-REFACTOR iterations to bulletproof
- Baseline testing revealed 10+ unique rationalizations
- Each REFACTOR closed specific loopholes
- Final compliance under maximum pressure: 100%

The same process works for any discipline-enforcing skill.
